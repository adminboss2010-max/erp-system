## 1. نظرة عامة على المشروع

نظام ERP (تخطيط موارد المؤسسة) متعدد المستأجرين (Multi-tenant)، وقابل للاستخدام من شركات أخرى.

**الريبو**: `adminboss2010-max/erp-system` على GitHub (عام/Public)
**الاستضافة**: GitHub Pages — `adminboss2010-max.github.io/erp-system/`
**الباك إند**: Google Apps Script (Code.gs) — **غير موجود على GitHub**، مربوط مباشرة بشيت "شركة لتجربة النظام" (Container-bound Script). أي تعديل عليه يتم يدويًا من محرر Apps Script + عمل Deploy جديد يدويًا.

### الصفحات الرئيسية
- `index.html` — تسجيل الدخول
- `app.html` — التطبيق الرئيسي (أكبر ملف، +10,700 سطر) — هنا كل شاشات العمل اليومي
- `admin.html` — لوحة الأدمن (إنشاء شركات جديدة، إلخ) — صغير نسبيًا (~300 سطر)، لا يحتوي أي منطق استيراد بيانات

### معمارية تعدد المستأجرين (Multi-tenant)
كل شركة جديدة تنضاف عبر `adminCreateCompany` في `Code.gs`، فتتولد لها:
- سطر في شيت "Companies" المركزي (Directory)
- Google Spreadsheet منفصل خاص بها فيه تابات: `المعاملات`, `العملاء`, `المناديب`, `الأصناف`, `بيانات الشركة`, `AppState` (مخفي), `AuditLog`

## 3. مشكلة تاريخية مهمة (تم حلها): عدم مزامنة البيانات بين المتصفحات

**العرض**: بيانات تُدخل من متصفح، تظهر فقط فيه، لا تظهر في متصفح آخر.

**السبب الجذري**: كود استرجاع البيانات من السحابة في `modules/30-isvaliddate.js` كان يُنفَّذ **مباشرة أثناء تحميل السكربت (synchronous)**، لكن `window.StorageV2` لا يُعرَّف إلا لاحقًا في `modules/33-infragenerateqr.js` (يُحمَّل بعده في `app.html`). النتيجة: الشرط `window.StorageV2` كان يفشل دائمًا، ومسار الاسترجاع من السحابة كان كودًا ميتًا.

**الحل المُطبَّق**:
- تأجيل الاسترجاع لحدث `DOMContentLoaded` بدل التنفيذ الفوري
- إضافة مطابقة زمنية (`_timestamp`) بين النسخة المحلية والسحابية بدل الاكتفاء بشرط "محلي فارغ"
- إضافة Debounce (900ms) على الحفظ السحابي لمنع إغراق السيرفر بطلبات متكررة
- معالجة تلقائية لتعارض الحفظ (Conflict) — سحب أحدث نسخة تلقائيًا بدل مجرد تحذير المستخدم

⚠️ **مهم جدًا**: هذا الحل تم تطبيقه **قبل** اكتشاف أن `45-server-only.js` موجود بالريبو وغير مفعّل، وأنه يطبّق فلسفة مختلفة تمامًا (منع التخزين المحلي نهائيًا بدل التوفيق بينه وبين السحابة). راجع القسم التالي.

## 4. قرار معماري حالي (قيد التنفيذ وقت كتابة هذا الملف): منع الاستيراد المحلي نهائيًا

**قرار صاحب المشروع**: البيانات يجب أن تأتي من السيرفر (Google Sheets عبر Code.gs) **فقط**، بدون أي استيراد محلي (Excel، JSON Backup) في أي صفحة، ومن غير أي كاش محلي (localStorage/IndexedDB) كمصدر حقيقة.

**الجرد الكامل لما يحتاج إزالة (تم في وقت كتابة هذا الملف)**:
- زر "استيراد ملف" + `<input type="file" id="importBackupInput">` في `app.html` (كان في نطاق الأسطر ~10729-10733 قبل أي تعديل، قد تتغير الأرقام بعد التعديلات)
- دالتا `importExcelFile()` و `handleExcelFile()` في `30-isvaliddate.js` — كانتا فعليًا **كود ميت** (لا يوجد عنصر `#excelFileInput` بالصفحة أصلًا يستدعيهما)
- `admin.html` و `index.html`: لا يوجد بهما أي استيراد بيانات، لا حاجة لتعديل

**القرار المعماري المتخذ**: بدل حذف كل إشارة يدويًا، الأفضل تفعيل `modules/45-server-only.js` (موجود جاهز ومصمم لهذا الغرض بالضبط — يعطّل كل عناصر الاستيراد عبر CSS selectors عامة، فيغطي أي إضافة مستقبلية مشابهة تلقائيًا). **يتطلب** دمج تحسينات الأداء (Debounce، معالجة التعارض التلقائية من القسم 3) داخل هذا الملف، لأنه يكتب فوق `StorageV2.save/load` الحالية.

**إذا كنت تقرأ هذا لاحقًا**: تحقق من `app.html` هل يوجد `<script src="modules/45-server-only.js">` أم لا، وهل `importJSONBackup`/`importExcelFile`/`handleExcelFile` ما زالت موجودة، لمعرفة إن كانت هذه الخطوة اكتملت أم لا.

## 5. أدوات MCP المتاحة (السيرفر المحلي بـ `D:\erp-mcp-server`)

⚠️ **مهم**: هذه الأدوات تعمل فقط داخل **Claude Code** (وليس عبر claude.ai في المتصفح). راجع `README_MCP_Project.md` بنفس الريبو أو مجلد المشروع المحلي للتفاصيل الكاملة (بيانات الاعتماد، خطوات التشغيل).

| الأداة | الوظيفة |
|---|---|
| `github_read_file` | قراءة ملف من الريبو |
| `github_list_files` | سرد الملفات داخل مجلد |
| `github_write_file` | كتابة/استبدال ملف كامل (⚠️ خطر مع الملفات الضخمة — يُفضّل `github_str_replace` بدلاً منه) |
| `github_str_replace` | استبدال جزء فريد ومحدد من ملف دون إعادة كتابته بالكامل — **الأداة المفضّلة للتعديلات على ملفات كبيرة مثل app.html** |
| `lint_js_file` | فحص Syntax لملف JS (مدمج تلقائيًا داخل أدوات الكتابة) |
| `sheets_list_tabs` | سرد أسماء التابات داخل Google Sheet |
| `sheets_read` | قراءة نطاق بيانات من تاب معين |
| `trace_dependencies` | البحث عن كل أماكن تعريف/استخدام دالة أو متغير عبر كل ملفات `modules/` — **الأداة الأهم لاكتشاف مشاكل ترتيب التحميل والتداخل** |

**قرار مؤجل**: أداة تحليل ملفات ضخمة عبر DeepSeek API — لم تُبنَ بعد، تُبنى فقط عند الحاجة الفعلية لتحليل ملف ضخم جدًا يصعب قراءته مباشرة.

## 6. مشاكل بيئة التطوير وحلولها (Windows + Claude Code)

| المشكلة | الحل |
|---|---|
| `Top-level await is currently not supported with the "cjs" output format` | `"type": "module"` في `package.json` |
| `running scripts is disabled on this system` (PowerShell) | `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned` |
| `cd D:\...` من C: لا ينقل الـ drive الفعلي | استخدم `cd /d D:\...` |
| أداة جديدة لا تظهر بعد تعديل الكود | لازم إغلاق Claude Code وإعادة فتحه (لا يعيد تحميل الكود تلقائيًا في نفس الجلسة) |
| `github_write_file` قد يفشل أو يتم رفضه من الـ agent مع ملفات ضخمة | استخدم `github_str_replace` بدلاً منه |

## 7. قرارات تم اتخاذها ورفضها (لتوفير وقت النقاش مستقبلًا)

- **لا هجرة لـ Supabase حاليًا**: المشروع بمرحلة تجربة، الهجرة الكاملة مشروع منفصل بحجم المشروع الأصلي، تُقيَّم لاحقًا عند الوصول لمرحلة إطلاق فعلي بعدد مستخدمين كبير.
- **لا "أدوات ذكاء" وهمية**: تم رفض فكرة بناء أدوات MCP مثل `analyze_module`, `explain_code`, `impact_analysis` كدوال منفصلة — هذه عمليات تحليل وفهم يقوم بها الـ AI مباشرة أثناء القراءة، وليست عمليات حتمية يمكن تحويلها لأداة. الأدوات تُبنى فقط للعمليات الحتمية (قراءة، كتابة، بحث، فحص).
- **لا نظام "طبقات ذكاء" ضخم (9 طبقات، +40 أداة)**: تم اقتراح رؤية طموحة جدًا لمنصة "ذكاء PROVA الكاملة"، وتقرر رفضها كخطة تنفيذ فعلية. البديل المتبع: بناء أداة جديدة فقط عند مواجهة حاجة حقيقية ملموسة، والاعتماد على هذا الملف (`AI_CONTEXT.md`) كطبقة "ذاكرة" بدل أدوات معقدة.
